# 十库跨主机同步一致性测试报告

- **测试时间**：2026-07-18
- **源端**：`127.0.0.1:3306`（MySQL 8.0.12）
- **目标端**：`192.168.88.88:3306`（MySQL 5.7.44，root/123456）
- **同步引擎**：remote-data-sync 内嵌引擎（`application.yml` 默认 profile，10 条 `127.0.0.1 → 192.168.88.88` 映射）
- **结论**：✅ 历史全量 + 实时增改删，10 个库数据全部一致

| 步骤 | 说明 |

## 二、过程中修复的关键问题

### 1. 目标 MySQL 5.7 连接数上限过低（导致同步/校验连不进）
- 目标默认 `max_connections=30`、`max_user_connections=25`，10 个跨主机作业即占满，外部校验触发 `1203 Too many connections` / `1040 Too many connections`。
- 临时调高（运行时，未持久化，重启目标 MySQL 后失效）：
  ```sql
  SET GLOBAL max_connections=200;
  SET GLOBAL max_user_connections=100;
  ```
- 校验脚本改为**单连接串行**（1 个源连接 + 1 个目标连接，通过 `USE db` 切换），避免“每库一连接”撑爆上限。

### 2. 跨时区 datetime 偏移 +15 小时（核心数据缺陷，已修复）
- **现象**：行数、金额汇总均一致，但 `create_time` / `update_time` 比源端多 15 小时。
- **根因**：Connector/J 8.0 的 `ResultSet.getObject()` 对 `DATETIME` 列返回 `java.time.LocalDateTime`（无时区）；
  `TableSinkFunction.setParam` 将其经 `Timestamp.valueOf(localDateTime)` 按 **JVM 默认时区（本机 PDT=UTC-7）** 转换，
  再经 `ps.setTimestamp` 按连接时区（Asia/Shanghai=UTC+8）写出，净偏移 = 8 − (−7) = **15h**。
- **修复**（4 处）：
  1. `TableSinkFunction.setParam`：`LocalDateTime` 改为 `ps.setObject(index, value)` 字面量绑定（时区无关）。
  2. `SyncDataBase.selectChanged`：增量水位 `WHERE update_time > ?` 改为 `ps.setObject(1, since)` 字面量绑定。
  3. `SyncDataBase`：水位更新兼容 `LocalDateTime` 类型（原仅认 `Timestamp`，导致增量水位从未刷新）。
  4. `RemoteDataSyncApplication.main`：启动设置 `TimeZone.setDefault(TimeZone.getTimeZone("Asia/Shanghai"))`，元数据时间戳也归中国时区。

## 四、遗留与建议
1. 目标 MySQL 连接上限为运行时调整，若 `192.168.88.88` 的 MySQL 重启需重新执行 `SET GLOBAL` 或写入 `my.cnf`（建议在目标机 `my.cnf` 增加 `max_connections=200`、`max_user_connections=100`）。
2. 原 `DRPlatformConsistencyTest`（JUnit）针对旧的本地 `geo/biz/cfg` 映射，现已不配置在默认 profile，重跑需先恢复对应映射或更新测试用例。
3. 后端服务运行中：`http://127.0.0.1:8080/`（看板可见 10 个跨主机同步作业状态）。
