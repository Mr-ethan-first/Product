- **日期**：2026-07-19
- **范围**：① 前端页面配置项与后端是否完全支持；② 异常场景 2.1~2.5（源/目标断电、动态新增库表、服务宕机重启、百万数据、其他）

## 一、前端 ↔ 后端配置一致性核对（需求①）

| 前端能力 | 前端字段（buildPairConfig / JSON） | 后端接收（SyncMappingRequestDTO） | 后端存储 / 返回（DatabaseMapping / VO） | 结论 |

| 通用忽略（DML+DDL，所有库） | `commonIgnoreTables` | `commonIgnoreTables` | `commonIgnoreTables` + `isDmlIgnored` | ✅ 已支持 |

> 后端补充动作：**无**（需求①已满足）。

## 二、同步引擎工作机制（理解容错的前提）

本系统的"实时同步"实质是**轮询快照同步**（非 binlog 流式 CDC）：

- 每个主机对启动一个 `SyncHostJob`（单线程定时任务，`pollIntervalMs=2000`）。
- 每轮：`getAllUserDatabases`（发现库）→ 逐库 `ensureTargetDatabase` → `listTables`（发现表）→ 逐表 `syncTable` → `reconcileDeletes`（删除对账）。
- 增量靠 `update_time` 水位（`selectChanged`）；首轮或重启后无水位 → 全量扫描。
- 写入用 `INSERT ... ON DUPLICATE KEY UPDATE`（幂等 upsert）；删除对账保证目标端不残留源端已删行。
- `SourceBinlogJob` 仅把源端 binlog 位点**记录到管控表**做台账，**并不用于恢复**；恢复依赖"重新全量扫描 + 水位"。
- 容错：`safeSync()` 整轮 `try/catch(Exception)`，单表失败不影响其他表；目标写入失败走 WAL 重试 → 死信队列（每 60s 重试）。

这套机制决定了"最终一致"的边界与下列场景结论。

## 三、异常场景测试结论

### 2.1 源数据库断电 / 断网 → 恢复后是否继续同步（最终一致）

- **机制**：`doSync()` 调用 `getAllUserDatabases(sourceHost)` 读取源端库清单。源端不可达时 `getAllUserDatabases` 捕获 `SQLException` 返回空列表；`safeSync` 不会抛错，**本轮空转，下一轮（2s 后）自动重试**。
- **结论**：✅ **支持**。源端恢复供电/网络后，下一个扫描周期即自动重新发现库表并继续同步，无需人工干预。
- **注意**：源端宕机期间发生的变更，在恢复后由"全量/增量重扫"补齐（`update_time` 水位 + 删除对账），保证最终一致。

### 2.1（续）目标数据库断电 / 断网 → 恢复后是否继续同步（最终一致）

- **机制**：
  - 库级：`ensureTargetDatabase(targetHost, db)` 失败 → 跳过该库，下一轮重试。
  - 表级：`ensureTargetTable` 失败 → 该表 sink 为空，跳过；恢复后重建。
  - 行写入：`TableSinkFunction` 先 WAL 重试 `maxWalRetries` 次，再降级单条，仍失败进**死信队列**每 60s 重试（本次新增**容量上限保护**，见第五节），避免长时不可用把 JVM 堆撑爆。
  - 背压：目标不可用时队列填满，`enqueue` 自旋等待，自然暂停源端读取，恢复后继续。
- **结论**：✅ **支持**。目标端恢复后死信队列与下一轮扫描共同补齐数据，最终一致。

### 2.2 源端动态新增库 / 新增表 → 是否动态同步

- **机制**：库清单与表清单**每轮重新发现**（`getAllUserDatabases` / `listTables`）。新库 → `ensureTargetDatabase` 在目标端建库；新表 → `DataBaseSinkFunction.ensureTable` 动态建 sink 并镜像建表；DROP 的表由 `knownTablesByDb` 比对后删除目标端对应表。
- **结论**：✅ **支持**。新增库/表在下一个扫描周期（≤`pollIntervalMs`）内被自动纳入，无需重启服务或改配置。
- **验证方式（可现场复现）**：在源端 `CREATE DATABASE new_db` / `CREATE TABLE` 后，观察看板"已配置主机对"下该库出现、目标端同名库被自动创建。

### 2.3 同步服务本身异常停机 / 断网 → 重启后是否动态继续（最终一致）

- **机制**：`@PostConstruct startSyncTask` 在启动时加载 `DRPlatform-dynamic-mappings.json`（页面动态新增的主机对）并启动全部 `SyncHostJob`；YAML 配置的主机对同样自动拉起。
- **关键点——重启后的水位**：`watermarks` / `knownSignatures` / `knownTablesByDb` 均为**内存态**，重启即清空 → 首轮退化为**全量重扫**。
  - 宕机期间源端的**插入/更新**：全量重扫 `SELECT *`（现已分页，内存恒定）全部补齐。
  - 宕机期间的**删除**：`reconcileDeletes` 比对源端 PK，删除目标端多余行。
  - 因此重启后**最终一致**。
- **结论**：✅ **支持**。服务重启、网络恢复后自动继续，无需人工触发。
- **性能提示**：首轮全量重扫对超大表较慢（但不再 OOM）。如需缩短恢复窗口，建议后续将水位持久化（见第六节建议）。当前"最终一致即可"目标下满足。

### 2.4 百万级数据同步 —— 是否存在数据丢失 / 不一致

**这是本次发现并修复的核心风险。**

- **原实现问题**：`syncTable` 走 `selectAll` → `queryRows` 执行 `SELECT * FROM db.table` 并**一次性把整表所有行读入 `List<Map<String,Object>>`**（无 LIMIT、无流式）。百万/千万行时：
  1. **直接 OOM（OutOfMemoryError）**；而 `safeSync` 只捕获 `Exception` 不捕获 `Error`，OOM 会**杀死调度线程 → 该主机对同步永久停止**；
  2. 重启后又走无 LIMIT 全量 → 再次 OOM → **同步永远起不来，目标端数据停滞 = 事实性数据丢失**；
  3. `reconcileDeletes` 同样把源端、目标端**全表 PK 各加载进一个 List**，百万行下同样撑爆堆。
- **本次修复**（详见第五节）：
  - 全量扫描改为 `streamScan`：**按单列主键 keyset 分页（无主键则 OFFSET 兜底）**，逐页处理，内存恒定（每页约 `batchSize*5` 行）；
  - 删除对账改为 `selectPksPaged` + `selectPksIn` 分页：目标端 PK 分页拉取，每页分批回查源端存在性；
  - 死信队列加容量上限，目标长时不可用时丢弃超出部分（依赖下一轮全量重扫补偿）。
- **修复后结论**：✅ **百万级数据安全**。无整表加载 → 不 OOM → 不丢数据；幂等 upsert + 删除对账保证不重不漏。
- **残留小边界（可接受）**：增量路径 `selectChanged` 用 `update_time > 水位` 且单批有 `LIMIT`；若同一 `update_time` 恰好跨批边界，极少数同时间戳行可能延到下一轮补齐（最终仍一致）。已由定期全量重扫兜底。

### 2.5 其他异常场景补充（最终一致）

| 场景 | 结论 | 说明 / 建议 |
|---|---|---|
| 源端 DDL（ALTER 加列）且未配置忽略 DDL | ✅ 支持 | `detectAndApplyAlter` 比对表结构指纹，变化时删目标表重建 + 清水位 → 重新全量同步该表 |
| 源端 DDL 且**已配置忽略 DDL**（`isDdlIgnored`） | ⚠️ 部分 | 目标端保持旧结构；若源端新增列，upsert 引用目标端不存在的列 → 写入失败进死信。**建议**：忽略 DDL 时应同时保证两端结构一致，或检测到列数不一致时自动触发该表重建 |
| 同步表**无主键** | ⚠️ 局限 | 无 PK 时 `reconcileDeletes` 跳过（不删目标多余行），upsert 退化为纯 INSERT → 重扫会产生重复行。**建议**：被同步表务必有主键 |
| 目标 5.7 不兼容源 8.0 的 DDL（除排序规则外） | ⚠️ 局限 | 排序规则已 `sanitizeDdlForPortability` 降级；但 generated column / invisible / descending index 等 8.0 专有语法仍会导致 `ensureTargetTable` 失败、该表被跳过。**建议**：灾备目标端尽量同大版本，或在建表语句归一化中扩展 |
| 连接池 / `max_user_connections` 打满 | ✅ 已缓解 | 全局信号量 `max-global-connections=200` + 每池 `max-pool-size=32`；目标端需 `SET GLOBAL max_connections/max_user_connections` 调高（见历史报告遗留项） |
| 进程被 kill / 正常退出 | ✅ 支持 | `@PreDestroy` 关闭前 flush 各表队列、关连接池；崩溃（kill -9）则依赖重启后全量重扫补偿 |
| 跨时区 datetime 偏移 | ✅ 已修复（历史） | `setParam` 对 `LocalDateTime` 用字面量 `setObject` 绑定，不按时区转换 |

## 五、本次后端改动清单（修复 2.4 / 加固 2.1·2.3）

**`sync/SyncHostJob.java`**
1. 新增 `streamScan(...)`：基于单列主键 keyset 分页（无主键 OFFSET 兜底）的流式全量扫描，内存恒定。
2. 新增 `queryPage(...)`：参数化分页查询。
3. 重写 `reconcileDeletes(...)`：分页拉取目标端 PK，分批（每 1000 一片）回查源端存在性，内存恒定。
4. 新增 `selectPksPaged(...)` / `selectPksIn(...)` 替换原一次性加载全表 PK 的 `selectPks`。
5. 拆分 `processRows(...)`：单页行处理（水位更新 / 字段转换 / 入队），供增量与全量共用。
6. 删除无 LIMIT 的 `selectAll`、无界 `selectPks`。

**`sink/TableSinkFunction.java`**
7. 死信队列 `pendingFailedEvents` 增加容量上限 `MAX_DEAD_LETTER=100_000`：超限丢弃并告警（依赖下一轮全量重扫补偿），防止目标端长时不可用时 OOM。

## 六、后续增强建议（非阻塞）

1. **水位持久化**：将 `update_time` 水位落库（或 Redis），重启后直接进入增量，避免每次重启全量重扫，缩短超大表的恢复窗口。
2. **忽略 DDL 时的结构自适应**：检测到源端列数变化且表被 DDL 忽略时，自动重建目标表（仍保留目标端数据需谨慎），或明确告警。
3. **无主键表策略**：提供"唯一索引兜底"或"整行哈希去重"的删除对账方案。
4. **目标端 DDL 归一化扩展**：在 `sanitizeDdlForPortability` 中补充 generated column / invisible / descending index 等 8.0→5.7 兼容处理。
5. **大表首轮限速**：超大表全量重扫可对单表限流，避免瞬时打满连接池与源端压力。
