## 1. 产品概述

### 1.1 产品定位

DRPlatform 是一个 **MySQL 跨主机数据库灾备同步中间件**，核心能力是将生产中心 MySQL 数据库的数据实时、可靠地同步到异地灾备中心。系统采用"主机对模型 + 自动库发现"的设计，运维人员只需配置一组源主机与目标主机，引擎即可自动发现源端全部用户库表并启动同步，新增库表无需人工介入。

### 1.2 问题陈述

传统数据库灾备方案存在以下痛点，DRPlatform 针对性解决：

| 痛点 | 传统方案影响 | DRPlatform 解决思路 |
|------|--------------|------|
| 新增库表易遗漏 | 业务系统持续新建库表，灾备配置需人工跟进 | 自动库发现（60 秒扫描周期），新增库表自动纳入同步 |
| 跨 MySQL 大版本 DDL 不兼容 | 8.0 专有 DDL 在 5.7 上执行报错 | DDL 清洗引擎自动降级为 5.7 兼容语法 |
| 配置需重启生效 | 修改忽略规则、字段映射需停服 | 配置热生效接口，不重启即时更新 |
| 故障恢复慢 | 断电断网后水位丢失，需人工重建 | 全量重扫 + 幂等 upsert 保证最终一致 |
| 缺乏可视化监控 | 排查问题依赖服务器日志 | Web 管理后台可视化监控同步进度与异常 |
| 忽略规则粒度不足 | 不能按库/表独立控制 DML 与 DDL | 双模式忽略规则，DML+DDL / 仅DDL 独立配置 |
| 操作无审计追溯 | 灾备配置变更无记录，事故难追责 | AOP 切面自动记录所有操作，密码脱敏后入库 |

### 1.3 非目标

- **不支持跨数据库类型同步**（MySQL → PostgreSQL 等），当前仅支持 MySQL → MySQL
- **不支持双向同步**，会引入冲突解决复杂度，非灾备场景核心诉求
- **不依赖 Flink 集群**，自研内嵌引擎 SyncHostJob，降低部署门槛
- **不使用 JWT/Token 鉴权**，管理后台场景下服务端 Session + Cookie 足够

## 2. 用户与场景

### 2.1 用户角色

| 角色 | 核心诉求 | 典型场景 |
|------|----------|----------|
| DBA / 运维工程师 | 巡检同步状态、新增灾备映射、快速定位异常 | 每日打开 Web 后台查看主机对状态与延迟，必要时一键重同步 |
| 开发 / 架构师 | 快速搭建异地灾备环境、验证数据一致性 | 初始化项目灾备配置，验证源/目标数据行数与字段一致性 |
| 项目经理 / 主管 | 掌握灾备覆盖率与整体状态、接收异常告警 | 查看仪表盘总览，关注异常主机对、同步延迟趋势 |

### 2.2 典型场景

**场景一：首次部署灾备链路**
运维人员登录系统 → 配置中心新增主机对（源 8.0、目标 5.7）→ 测试连通性 → 保存 → 引擎自动启动全量同步 → 在统计监控页查看进度与延迟。

**场景二：业务侧新增数据库**
业务系统在源 MySQL 上新建库与表 → 60 秒内 ScanNewDatabaseJob 自动发现 → 引擎自动启动该库全量同步 → 全程无需运维介入。

**场景三：断电断网恢复**
服务进程重启或网络中断恢复后 → 水位表读取失败 → 引擎对受影响表执行全量重扫 → 幂等 upsert 保证最终一致 → 同步延迟恢复至 < 3 秒。

**场景四：排除系统库表同步**
运维在主机对配置中为 DML 与 DDL 分别配置忽略规则（如 `mysql.*`、`sys.*`、`information_schema.*`）→ 引擎按规则跳过对应库表的 DML/DDL 操作。

### 3.3 关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 不使用 Flink 集群 | 自研内嵌引擎 SyncHostJob | 降低部署复杂度，无需额外组件 |
| 前端不做 Node 构建链 | React UMD + 构建期预编译 app.js | 避免运行时 Babel 转译浏览器报错 |
| 死信队列上限 100,000 | 超限丢弃 + 全量重扫补偿 | 防止 OOM，保证最大恢复 |
| 目标删除使用 PK WHERE | 非 `ON DUPLICATE KEY UPDATE DELETE` | 跨版本兼容（5.7 不支持该 DELETE 语法） |
| 元数据库独立 | `DRPlatform` 与业务库分离 | 配置与水位不受业务库影响 |

## 4. 功能需求

### 4.1 主机对自动库发现

**业务逻辑**
系统以"主机对"为基本管理单元，一组主机对包含源端 MySQL 连接信息与目标端 MySQL 连接信息。引擎启动后自动调用 `information_schema.SCHEMATA` 与 `information_schema.TABLES` 发现源端全部用户库（排除 `mysql`、`sys`、`information_schema`、`performance_schema` 等系统库），并对每个库的全部用户表启动同步任务。

**新增库表自动纳入**
后台 `ScanNewDatabaseJob` 每 60 秒触发一次扫描：
1. 遍历所有启用状态的主机对
2. 重新拉取源端库表清单
3. 与本地元数据库中已记录的库表对比
4. 对新增的库表自动创建同步任务并启动全量同步

**交互流程**
- 用户在 Web 后台新增主机对 → 填写源/目标连接参数 → 点击"测试连接"校验可达性 → 保存
- 引擎立即触发一次发现 → 同步任务列表实时刷新，可见新纳入的库表
- 用户可在"同步进度"页查看每个库表的同步状态

**关键规则**
- 系统库（`mysql`、`sys`、`information_schema`、`performance_schema`）永不纳入同步
- 主机对支持启用/停用状态切换，停用后所有相关同步任务暂停
- 连接信息中的密码在入库时加密存储，在 API 返回时脱敏
- 同一源主机可被多个主机对引用，但每个主机对的目标主机必须独立

### 4.2 全量 + 增量同步

**全量同步**
首次同步或水位丢失时执行全量流式分页扫描：
1. 按 `LIMIT offset, size` 分页拉取源表数据，分页大小可配置（默认 1000 行/页）
2. 每页拉取后立即批量写入目标端（`INSERT ... ON DUPLICATE KEY UPDATE` 幂等 upsert）
3. 单页处理完成后立即释放内存，保证内存占用恒定，不随表大小增长
4. 全量完成后写入初始水位（`watermarks` 表）

**增量同步**
对含 `update_time` 列的表启用增量模式：
1. 读取上次同步水位 `watermarks.last_sync_value`
2. 仅扫描 `WHERE update_time > ?` 的增量数据
3. 增量数据 upsert 完成后更新水位
4. 周期性触发（默认 5 秒），保证 P99 延迟 < 3 秒

**交互流程**
- Web 后台"同步进度"页展示每个库表的同步模式（全量/增量）、当前水位、偏差时间
- 用户可手动触发"重新同步"，强制对该表执行全量重扫

**关键规则**
- 无 `update_time` 列的表只支持全量模式，每次扫描为全表扫描
- 全量同步采用流式分页，禁止一次性 `SELECT *` 全量加载，避免 OOM
- 增量水位采用"先写目标后更新水位"的顺序，确保幂等可重入
- 同一张表同一时刻仅允许一个同步任务执行（基于表级锁）

### 4.3 双模式忽略规则

**业务逻辑**
忽略规则支持 DML 与 DDL 两类操作的独立控制：
- **DML+DDL 忽略**：跳过该库表的所有数据变更（INSERT/UPDATE/DELETE）和结构变更（ALTER/DROP/CREATE）
- **仅 DDL 忽略**：跳过结构变更，但保留数据行同步

两种模式可独立配置在库粒度或表粒度上，互不干扰。

**匹配模式**
支持三种匹配语法，可在同一规则集中混用：
- **精确匹配**：`db_name.table_name` 完全匹配
- **Glob 通配**：`db_*.log_*` 支持通配符
- **正则匹配**：`^shard_\d+\..*` 标准正则

**交互流程**
- 用户在主机对配置页填写忽略规则，DML 与 DDL 两个独立文本框
- 保存后通过配置热生效接口即时更新引擎，无需重启
- 引擎在每次执行同步前先匹配忽略规则，命中则跳过

**关键规则**
- 忽略规则按主机对隔离，不同主机对规则独立
- 库粒度规则优先生效，若库被忽略则该库下所有表不再逐表判断
- 规则匹配失败（如正则语法错误）时降级为不忽略，但记录告警日志
- 忽略规则变更后，已被同步的历史数据不会回滚，仅影响后续同步行为

### 4.4 跨版本 DDL 清洗

**业务逻辑**
源端为 MySQL 8.0、目标端为 MySQL 5.7，存在大量语法不兼容的 DDL（如 ` Invisible`、`utf8mb4_0900_ai_ci`、`CHECK` 约束等）。引擎在执行 DDL 前调用 `sanitizeDdlForPortability` 清洗函数，将 8.0 专有语法降级为 5.7 兼容语法。

**清洗规则集**

| MySQL 8.0 语法 | 降级处理 |
|--------------|----------|
| `CHARACTER SET utf8mb4 COLLATE utf8mb4_0900_ai_ci` | 替换为 `utf8mb4_general_ci` |
| `INVISIBLE` 列属性 | 移除该关键字 |
| `CHECK (expr)` 约束 | 移除该约束声明 |
| `DEFAULT (expr)` 表达式默认值 | 替换为字面值或移除 |
| `ENGINE = InnoDB ROW_FORMAT=DYNAMIC` | 保留，5.7 兼容 |
| `AUTO_INCREMENT = N` | 保留 |
| `IF NOT EXISTS` / `IF EXISTS` | 保留，5.7 兼容 |
| 8.0 关键字 `JSON_TABLE`、`WITH` 等 | 转换或移除 |

**交互流程**
- 用户配置主机对时自动识别源端版本（`SELECT VERSION()`）
- 引擎在每次执行 ALTER / CREATE 前调用清洗函数
- 清洗前后的 DDL 文本记录在审计日志中，便于追溯

**关键规则**
- 清洗函数幂等，对已兼容的 DDL 不会产生副作用
- 清洗失败的 DDL 进入死信队列，不阻塞后续同步
- 清洗规则集可通过配置文件扩展，无需修改代码
- 对无法自动降级的语法（如 `JSON_TABLE`），记录 WARN 日志并跳过该 DDL

### 4.5 字段转换

**业务逻辑**
在 upsert 写入目标端前，对每行数据按字段映射规则进行值变换。字段映射分为两类：
- **全局映射**：在 `application.yml` 中配置，对所有用户可见且生效，适合跨环境通用规则（如 IP 段替换、全局脱敏策略）
- **私有映射**：在 Web 后台为单主机对配置，仅创建者可见，适合特定业务的特殊处理

**支持的转换类型**

| 转换类型 | 适用场景 | 示例 |
|----------|----------|------|
| IP 替换 | 灾备环境网段与生产不同 | `10.0.1.23` → `192.168.1.23` |
| 字段脱敏 | 灾备环境隐藏敏感信息 | 手机号 `138****1234` |
| 枚举映射 | 不同环境的字段值差异 | `prod` → `dr` |

**交互流程**
- 用户在主机对详情页配置字段映射规则（表名、字段名、转换类型、源值→目标值）
- 保存后通过配置热生效接口即时更新
- 引擎在 upsert 前按字段映射逐行变换

**关键规则**
- 全局映射优先级低于私有映射，同字段冲突时私有映射覆盖
- 转换失败的字段保留原值并记录 WARN 日志，不中断同步
- 字段映射基于字段名匹配，不依赖字段顺序
- 支持同一字段配置多条映射规则，按声明顺序应用

### 4.6 Web 管理后台

**业务逻辑**
基于 React 18 UMD 构建的单页应用，提供可视化配置与监控能力，覆盖主机对管理、同步进度监控、忽略规则配置、字段映射、操作日志、用户管理等模块。前端通过 Nginx 反向代理访问后端 REST API，鉴权基于服务端 Session + Cookie。

**主要功能模块**

| 模块 | 核心功能 |
|------|----------|
| 仪表盘 | 主机对总数、正常/异常数、同步覆盖率、平均延迟等关键指标汇总 |
| 同步进度 | 每个库表的当前状态、同步模式、水位、偏差时间，支持筛选与一键重同步 |
| 主机对管理 | 增删改查主机对、测试连接、配置忽略规则与字段映射 |
| 操作日志 | 全量操作审计记录，支持按用户、类型、时间范围筛选 |
| 用户管理 | 账号注册、登录、登出、权限隔离 |

**交互流程**
- 用户访问 `/` 被重定向至登录页 → 输入账号密码 → 服务端创建 Session → 跳转仪表盘
- 仪表盘默认 5 秒轮询刷新关键指标
- 同步进度页支持按主机对、库、表名筛选，异常项置顶高亮
- 主机对详情采用右侧抽屉面板，无需跳转页面即可编辑配置

**关键规则**
- 未登录用户访问任意接口返回 401，前端跳转登录页
- Session 默认有效期 30 分钟，可通过配置调整
- 所有写操作（POST/PUT/DELETE）必须携带 CSRF Token
- 前端无 Node 构建链，JSX 在构建期预编译为 `app.js`，运行时无 Babel 依赖
- 响应式布局，移动端表格横向滚动，抽屉面板全屏适配

### 4.7 多用户隔离

**业务逻辑**
系统支持多账号体系，配置数据按用户隔离：
- **私有配置**：用户创建的主机对、忽略规则、私有字段映射仅对创建者可见，其他用户无法查看或操作
- **全局配置**：通过 `application.yml` 配置的全局字段映射对所有用户可见，但不可在 Web 后台修改
- **共享数据**：操作日志、同步进度等运行时数据对管理员可见，普通用户仅可见自己创建的资源

**交互流程**
- 用户登录后，所有列表查询自动追加 `owner_id = current_user_id` 过滤条件
- 管理员账号可查看全部用户的配置（用于运维排查），但不可修改他人配置
- 用户注销后其私有配置可由管理员迁移或删除

**关键规则**
- 私有字段映射与全局字段映射冲突时，私有映射优先级更高
- 删除用户账号时支持级联删除其私有配置，或转移至其他用户
- 用户角色分两类：普通用户、管理员，管理员可管理所有账号
- 多用户共享同一元数据库，通过 `owner_id` 字段隔离

### 4.8 操作审计日志

**业务逻辑**
基于 Spring AOP 切面自动拦截所有 Controller 方法，将操作上下文持久化到 `operation_log` 表，无需在业务代码中手动埋点。

**审计字段**

| 字段 | 说明 |
|------|------|
| `id` | 主键自增 |
| `user_id` | 操作用户 ID，未登录操作为空 |
| `username` | 操作用户名 |
| `client_ip` | 客户端 IP，从 `X-Forwarded-For` 或 `RemoteAddr` 提取 |
| `operation_type` | 操作类型枚举（LOGIN / LOGOUT / REGISTER / SYNC_CONFIG / RESYNC 等） |
| `request_params` | 请求参数 JSON，密码类字段自动脱敏（如 `password: "***"`） |
| `execution_result` | 执行结果（SUCCESS / FAILURE） |
| `error_message` | 失败时的错误信息 |
| `duration_ms` | 操作耗时（毫秒） |
| `created_at` | 操作时间戳 |

**交互流程**
- 所有接口被 AOP 切面自动包裹，无需业务侧显式调用
- Web 后台"操作日志"页提供分页查询，支持按用户、类型、时间范围、结果筛选
- 管理员可导出日志为 CSV 用于合规审计

**关键规则**
- 密码、密钥、Token 等敏感字段在入库前必须脱敏，配置在 `sensitive.fields` 列表中
- 日志表按月分区或定期归档，避免单表过大影响查询性能
- 审计日志写入失败不阻塞主业务，仅记录 ERROR 级别告警
- 日志保留期默认 90 天，可通过配置调整

### 4.9 配置热生效

**业务逻辑**
通过 `/sync/mapping/update` 与 `/sync/mapping/reload` 两个接口实现配置动态更新，无需重启服务：
- **update**：增量更新单个主机对的配置（忽略规则、字段映射、连接参数等）
- **reload**：全量重载所有主机对配置，用于配置文件或数据库被外部修改后的同步

**交互流程**
- 用户在 Web 后台修改配置 → 点击保存 → 调用 `/sync/mapping/update`
- 引擎在内存中更新对应主机对的配置对象 → 通知运行中的同步任务重新读取
- 同步任务在下一次循环开始前应用新配置，不影响当前正在执行的批次

**关键规则**
- 配置更新为原子操作，要么全部应用成功，要么全部回滚
- 更新失败时返回详细错误信息（如正则语法错误、字段名不存在等）
- `reload` 操作会暂停所有同步任务，重载完成后恢复
- 配置热生效不影响水位数据，已同步的进度不会丢失
- 配置变更记录自动写入操作审计日志

### 4.10 断电断网续传

**业务逻辑**
针对断电、断网、进程崩溃等异常场景，保证同步最终一致：

1. **水位丢失处理**：服务重启后读取 `watermarks` 表，若水位记录不存在或无法解析 → 对受影响表执行全量重扫 + 幂等 upsert 重建水位
2. **连接临时不可用**：HikariCP 自动重连，同步任务在重试窗口内（默认 30 秒）等待恢复；超时后进入死信队列
3. **死信队列补偿**：失败的操作写入死信表，上限 100,000 条，超限丢弃并触发全量重扫
4. **幂等 upsert**：所有写入目标端采用 `INSERT ... ON DUPLICATE KEY UPDATE`，重复执行不会产生副作用

**交互流程**
- 服务启动时检查上次水位完整性 → 不完整则自动触发全量重扫
- 同步过程中目标端不可达 → HikariCP 进入重试 → 30 秒内恢复则继续，否则入死信队列
- Web 后台"同步进度"页展示死信队列长度，支持手动触发重扫

**关键规则**
- 故障恢复时间目标 < 30 秒（HikariCP 重连 + 任务恢复）
- 死信队列超限时优先丢弃最旧记录，保证不 OOM
- 全量重扫期间该表不接受增量更新，重扫完成后合并增量
- 水位更新采用"先写目标后更新水位"顺序，确保崩溃后可重入
- 进度数据持久化到元数据库，重启后自动加载 `dynamic-mappings`

### 4.11 DDL 自动同步

**业务逻辑**
引擎自动检测源端结构变更并同步到目标端：

1. **ALTER 检测**：通过表结构指纹比对（字段类型、长度、索引、约束的哈希值）发现源端结构变更 → 自动生成兼容 5.7 的 ALTER DDL → 在目标端执行
2. **DROP 检测**：扫描源端库表清单，若某表在源端消失（被 DROP）→ 在目标端执行 `DROP TABLE` 删除对应表
3. **CREATE 检测**：发现源端新增表 → 自动在目标端创建对应表结构（DDL 经清洗）

**交互流程**
- 引擎周期性（默认 60 秒）比对源/目标端表结构
- 检测到变更后自动应用 DDL（经清洗），记录在审计日志
- Web 后台"同步进度"页展示 DDL 操作历史

**关键规则**
- DDL 操作前必须经跨版本清洗（见 4.4）
- DROP 操作不可逆，执行前记录详细日志，便于回滚追溯
- 受"仅 DDL 忽略"规则匹配的表不执行任何 DDL
- 受"DML+DDL 忽略"规则匹配的表不执行 DDL 也不执行 DML
- DDL 执行失败进入死信队列，不阻塞后续表

### 4.12 删除对账

**业务逻辑**
`reconcileDeletes` 任务定期扫描目标端，删除目标端存在但源端不存在的多余行，保证双向最终一致。这是为了处理源端执行 `DELETE` 后目标端未能及时同步的情况（如断网期间丢失的 DELETE 操作）。

**对账策略**
1. 对每张表按主键分页比对源/目标端
2. 目标端存在但源端不存在的行 → 执行 `DELETE FROM target WHERE pk = ?`
3. 大表首次全量对账跳过（避免长时间占用资源），仅对增量范围对账
4. 对账完成后记录对账结果（删除行数、耗时）

**交互流程**
- 引擎周期性触发对账（默认每日凌晨低峰期）
- Web 后台展示对账历史与删除行数
- 用户可手动触发对账，强制执行全量比对

**关键规则**
- 对账删除基于主键匹配，不依赖 `update_time` 列
- 大表首次全量对账默认跳过，避免长时间锁表
- 对账期间该表不接受其他写操作，避免数据竞争
- 对账删除采用批量 DELETE，单批默认 1000 行
- 对账结果记录在审计日志，便于追溯

### 4.13 HA 高可用

**业务逻辑**
针对 MySQL 主备切换场景提供 HA 检测与自动恢复能力：

1. **DBHASwitchStatusJob**：周期性检测源端 MySQL 是否发生主备切换（通过 `SHOW MASTER STATUS` 或 `server_id` 变化）
2. **RestartTaskManager**：管理重启任务队列，主备切换后自动重启受影响的同步任务
3. **连接池重建**：主备切换后 HikariCP 自动销毁旧连接，建立到新主库的连接

**交互流程**
- 默认配置为关闭状态，需在 `application.yml` 显式启用
- 启用后每 30 秒检测一次主备状态
- 检测到切换后暂停所有同步任务 → 等待新主库可用 → 重建连接池 → 恢复同步任务

**关键规则**
- HA 检测为可选功能，默认关闭，避免对未启用主备架构的环境造成干扰
- 切换检测窗口期内同步任务暂停，数据不丢失（源端 Binlog 持久化）
- 切换完成后自动触发一次全量重扫，确保新主库数据完整同步
- 切换事件记录在审计日志，包含切换时间、旧/新 server_id

## 7. 技术风险与缓解

| 风险 | 可能性 | 缓解措施 |
|------|--------|----------|
| MySQL 8.0→5.7 DDL 仍有未覆盖的不支持语法 | 中 | 持续维护 `sanitizeDdlForPortability` 规则集，未覆盖 DDL 入死信队列并告警 |
| 单机性能瓶颈（连接数/线程数） | 低 | 通过 `DRPlatform.connection.max-global-connections` 调节全局信号量限流 |
| 大表全量同步占用资源过久 | 中 | 流式分页 + 增量水位扫描，大表首次全量对账跳过 |
| 死信队列积压影响恢复 | 低 | 上限 100,000 条，超限触发全量重扫补偿 |
